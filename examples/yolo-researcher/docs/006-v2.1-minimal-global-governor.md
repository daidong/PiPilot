# YOLO-Researcher v2.1: Minimal Global Governor

> **Design Axiom**: The system does not pursue quality through architectural complexity.
> It pursues **minimal discipline to avoid death + evidence-driven progressive strengthening**.
>
> This document extends `005-v2-minimal-reliable.md` with one missing piece:
> a minimal global governance layer to prevent plan drift.

---

## 0. Why v2.1 Exists

`v2` solved major reliability issues (append-only runs, evidence pointers, failure memory), but it still drifts on open-ended research tasks:

- turn-by-turn local optimization dominates
- plan text changes every turn
- unfinished work silently disappears
- "success" can mean "new artifact exists" without real mainline progress

This is not a reason to return to v1 heavy multi-role orchestration.
It is a reason to add one thin governance mechanism.

---

## 1. Decision

Use:

- **Single execution agent** (native tools/skills/subagents)
- **Sparse planner checkpoint** (global board maintenance only)

Do **not** reintroduce:

- mandatory stage pipeline (S1-S5 gating)
- reviewer gates as correctness dependencies
- heavy coordinator/planner/executor protocols

---

## 2. Non-Negotiable Principles

1. **Single Executor Stays**
   One execution agent does real work. Planner is not an execution lane.

2. **Plan Is an Object, Not Narrative Text**
   Every mainline task has a stable ID. IDs do not change when wording changes.

3. **Progress Must Bind to a Plan ID**
   Turn success is valid only when it advances one active plan item with evidence.

4. **Carry-Forward by Default**
   Unfinished items persist automatically. No silent dropping.

5. **Append-Only Evidence**
   Facts and constraints still require `runs/turn-xxxx/...` pointers.

---

## 3. Minimal Architecture

```text
Turn Loop
  ├─ Execute turn with single native agent
  ├─ Bind turn to one active_plan_id
  ├─ Classify result mechanically (success/no_delta/failure/blocked/ask_user/stopped)
  ├─ Persist artifacts + pointers
  └─ Occasionally run planner checkpoint (sparse trigger)
```

Planner checkpoint is governance only:

- reorder/prioritize plan board
- refine done definitions
- maintain top-3 and backlog

It does not replace execution decisions.

---

## 4. Plan Board Contract (in PROJECT.md)

No new storage system is required.
Add one section in `PROJECT.md`:

```markdown
## Plan Board (stable IDs)

### Active (WIP limit = 1)
- P2 [ACTIVE] Build hookpoint map artifact
  done_definition:
  - artifacts/hookpoints.md exists
  - each hook has evidence pointer under runs/turn-xxxx/
  evidence:
  - runs/turn-0014/artifacts/excerpt_process_parallel.md
  next_min_step:
  - extract evaluator cascade promotion hook and add pointer

### Top-3 Next
- P3 [TODO] Literature matrix (themes x papers)
  done_definition:
  - artifacts/literature/matrix.md exists with >= 12 de-duplicated papers

- P4 [TODO] Minimal telemetry patch
  done_definition:
  - patch.diff exists + execution evidence confirms telemetry fields emitted

### Backlog
- P5 [TODO] Draft positioning section

### Done (Do-not-repeat)
- FETCH_UPSTREAM:controller/database/evaluator/prompt_sampler/config
  evidence: runs/turn-0005/artifacts/upstream_index.md
```

### 4.1 Stable ID Rules

- IDs are monotonic: `P1`, `P2`, `P3`, ...
- IDs are never reused
- renaming an item does not change its ID

### 4.2 WIP Limit

- exactly one `ACTIVE` item at a time (default)
- optional temporary override only with explicit reason in `action.md`

---

## 5. Turn Contract Additions

Each turn must include:

- `active_plan_id`
- `status_change` (e.g. `P2: ACTIVE 30% -> 50%`, or `TODO -> ACTIVE`, or `ACTIVE -> DONE`)
- `delta` (what changed)
- `evidence` pointer(s)

Suggested `action.md` delta block:

```markdown
## Plan Delta
- active_plan_id: P2
- status_change: P2 ACTIVE 30% -> 50%
- delta: updated artifacts/hookpoints.md with evaluator cascade hook
- evidence: runs/turn-0021/artifacts/excerpt_evaluator.md
```

---

## 6. Mechanical Status Semantics

Allowed turn statuses:

- `success`
- `no_delta`
- `failure`
- `blocked`
- `ask_user`
- `stopped`

Classification policy:

```text
if missing active_plan_id:
  no_delta
else if deterministic execution failure:
  failure (or blocked via failure circuit)
else if asks for required external input:
  ask_user
else if evidence-backed progress on active_plan_id done_definition:
  success
else:
  no_delta
```

Important:

- New artifact without active plan advancement = `no_delta`
- "Looks useful" narrative without evidence pointer = `no_delta`

---

## 7. Carry-Forward and Drop Discipline

### 7.1 Default Carry-Forward

- unfinished `TODO/ACTIVE/BLOCKED` items must remain on board
- planner/executor cannot silently remove them

### 7.2 Drop Requires Explicit Record

To remove an item, write:

- `drop_reason` (one sentence)
- `evidence` pointer
- `replaced_by: Pn | null`

If missing any field, drop is invalid and item must stay.

---

## 8. Redundancy Breaker for "Repeated Success"

Failure breaker handles repeated errors.
v2.1 adds minimal redundancy breaker for repeated non-progress.

Fingerprint (mechanical, non-semantic):

```text
fingerprint = action_type + normalized_target
```

Rule:

- if same fingerprint repeats and current turn is `no_delta`
- append `[BLOCKED][REDUNDANT]` to `FAILURES.md`
- next turn must change artifact type or target plan item

This prevents "successful busy loops".

---

## 9. Sparse Planner Checkpoint

Planner checkpoint runs only when triggered:

- every 3-4 turns
- two consecutive `no_delta`
- redundancy blocker triggered
- top-3 all blocked
- user goal/constraints changed

Planner output is strictly limited to:

1. maintain `Plan Board` (IDs/state/priority)
2. refine `done_definition`
3. choose next `ACTIVE` item

Planner checkpoint must not:

- execute tools
- rewrite facts/constraints narrative wholesale
- introduce heavy stage protocol

## 10. File Contract Changes from v2

### 10.1 PROJECT.md

Add/maintain:

- `Plan Board (stable IDs)`
- `Done (Do-not-repeat)`

Existing sections from v2 remain valid.

### 10.2 action.md

Add required `Plan Delta` block fields:

- `active_plan_id`
- `status_change`
- `delta`
- `evidence`

### 10.3 FAILURES.md

Allow redundant-progress memory entry:

```markdown
- [BLOCKED][REDUNDANT] <fingerprint>
  reason: repeated action without plan-linked delta
  evidence: runs/turn-00xx/action.md
  updated_at: <iso>
  alternatives:
    - switch active_plan_id
    - produce new artifact type
```

No other new file is required.

---

## 11. Acceptance Criteria (Observable in Logs)

v2.1 is considered healthy when:

1. Consecutive turns usually keep the same `active_plan_id` until done or blocked.
2. Plan board changes are incremental, not wholesale rewrites every turn.
3. Unfinished items do not disappear silently.
4. `success` turns always carry evidence-backed progress on a plan item.
5. Repeated "safe but useless" loops trigger `[BLOCKED][REDUNDANT]`.

---

## 12. Why This Still Honors v2 Philosophy

- **No heavy architecture comeback**: still single executor, thin contracts.
- **Minimal discipline only**: one plan board, one active item, sparse checkpoint.
- **Evidence-driven strengthening**: progress defined by evidence against done criteria.
- **Not fragile, not over-engineered**: enough structure to stop drift, no protocol bloat.

## Appendix A: Minimal Planner Checkpoint Prompt Skeleton

```text
You are the v2.1 planner checkpoint (governance only).

Inputs:
- current PROJECT.md plan board
- recent turn outcomes
- FAILURES.md

Rules:
- Keep stable IDs, never reuse IDs
- Do not drop unfinished items silently
- Select exactly one ACTIVE item (unless blocked)
- Update done_definition only when clearer/measurable
- Keep Top-3 focused; move others to backlog
- Output concise board update only (no execution plan prose)
```

---

## Appendix B: Minimal Executor Prompt Addendum

```text
Before acting, choose one active_plan_id from Plan Board.
Your turn is success only if you produce evidence-backed progress for that plan item.
If output is useful but not advancing active_plan_id, mark no_delta.
Do not silently drop unfinished items.
```
