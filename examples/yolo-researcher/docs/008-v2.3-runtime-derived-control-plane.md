# YOLO-Researcher v2.3: Runtime-Derived Control Plane (RFC-008)

> Goal: keep strict convergence discipline while removing fragile LLM-authored control fields.
> The runtime, not the model, is the source of truth for turn control metadata.

---

## 0. Why v2.3 Exists

`v2.1` and `v2.2` added the right guardrails, but they still rely on LLM-authored control fields:

- `activePlanId`
- `statusChange`
- `delta`
- `evidencePaths`
- drop metadata (`dropReason`, `replacedBy`)

When any one field is malformed or references a non-existent path, turns can degrade into repeated `no_delta` even if useful artifacts were produced.

v2.3 removes this fragility by shifting control-plane authorship from LLM output to deterministic runtime derivation.

---

## 1. Decision

1. LLM emits only minimal intent-level output.
2. Runtime auto-generates control-plane fields from tool events + filesystem writes.
3. Plan structure edits remain gated by planner checkpoint windows.
4. Success requires deliverable touch (or blocker clear), evaluated mechanically by runtime.
5. Facts/claims evidence linkage is runtime-attached (bundle or refs), not path-authored by LLM.

This is a simplification, not an expansion of constraints.

---

## 2. Non-Negotiable Invariants

1. Evidence existence:
   All evidence pointers must resolve to existing files under `runs/turn-xxxx/...`.
2. Plan binding:
   A turn classified as `success` must bind to one active plan item and have plan delta.
3. No-progress breaker:
   repeated `no_delta` triggers blocker/escalation flow.

These stay unchanged from v2.1/v2.2.

---

## 3. LLM Output Contract (Reduced)

New minimal turn outcome contract:

```json
{
  "intent": "why this turn",
  "summary": "concise observation",
  "statusHint": "success|failure|ask_user|stopped",
  "primaryAction": "optional short label",
  "askQuestion": "required if statusHint=ask_user",
  "stopReason": "required if statusHint=stopped",
  "projectUpdate": {
    "goal": "... optional ...",
    "successCriteria": ["... optional ..."],
    "planBoard": ["... optional, checkpoint only ..."],
    "currentPlan": ["... optional, checkpoint only ..."],
    "facts": ["... optional ..."],
    "constraints": ["... optional ..."],
    "hypotheses": ["... optional ..."],
    "claims": ["... optional ..."]
  },
  "rawOutput": "optional"
}
```

Removed from LLM contract:

- `activePlanId`
- `statusChange`
- `delta`
- `evidencePaths`
- `dropReason`
- `replacedBy`

Those fields are runtime-derived only.

---

## 4. Runtime-Derived Control Fields

For each turn, runtime derives:

1. `activePlanId`
2. `statusChange`
3. `delta`
4. `planEvidencePaths`
5. `turnStatus` (final)

### 4.1 Deriving `activePlanId` (Post-Turn Attribution)

`activePlanId` is derived after tool execution, not before:

1. compute `touched_files` from turn artifacts plus workspace writes (`changed_files.json`, optional `patch.diff`)
2. parse each plan item's mechanical deliverables list
3. compute `match_score = touched_files ∩ deliverables`
4. if exactly one best-matching plan exists, bind to that plan
5. if multiple plans match in the same turn:
   classify as `no_delta` and trigger planner checkpoint (split required)
6. if no deliverable match, but blocker was cleared, bind to blocker owner plan
7. only then fallback to current `ACTIVE`, else Top-3 highest-priority `TODO`
8. if still none, classify `no_delta`

### 4.2 Deriving `delta`

Runtime computes delta from:

- successful command execution bundle
- artifact file write set in this turn
- workspace write set in this turn (`changed_files.json`, optional `patch.diff`)
- blocker transitions (`BLOCKED` -> `UNBLOCKED`)

No natural-language delta from LLM is trusted.

### 4.3 Deriving `planEvidencePaths`

Runtime unions existing canonical paths from:

- implicit turn files:
  - `cmd.txt`
  - `stdout.txt`
  - `stderr.txt`
  - `exit_code.txt`
  - `result.json`
  - `artifacts/tool-events.jsonl`
  - `artifacts/changed_files.json`
  - `artifacts/patch.diff` (when repository writes exist)
- business artifacts created this turn under `artifacts/`

Only existing canonical paths survive.

### 4.4 Deriving `statusChange`

Rules:

1. default: `Pn ACTIVE -> ACTIVE`
2. if no `ACTIVE` existed and `TODO` promoted: `Pn TODO -> ACTIVE`
3. if done_definition satisfied cumulatively: `Pn ACTIVE -> DONE`
4. if deterministic blocker escalation: `Pn ACTIVE -> BLOCKED`

Drop transitions (`-> DROPPED`) are planner-checkpoint-only and must come from validated structured plan edits, not free-form turn output.

### 4.5 Evidence Bundling for Facts/Constraints/Claims

Runtime writes one evidence bundle per turn in `result.json`:

- `evidence_paths: [...]` (resolved canonical paths)

For any LLM-proposed `facts/constraints/claims`:

1. if row references runtime `E#` tokens, resolve `E# -> canonical path`
2. if no evidence ref is provided, auto-attach `evidence: runs/turn-xxxx/result.json`
3. if attach fails, downgrade row to `[HYP]` and do not treat as fact/claim evidence

LLM never authors raw evidence paths.

### 4.6 Mechanical Deliverable Contract

Each plan item must use machine-checkable done rules:

- `deliverable: <path-or-token>` (required, one or more)
- `evidence_min: <n>` (optional)

Natural-language done definitions are non-compliant and treated as unmet.

---

## 5. Final Status Classification (Runtime)

`statusHint` is advisory only.
Runtime computes final status:

```text
if ask-user gating condition:
  ask_user
else if deterministic execution failure and not remediated:
  failure or blocked
else if no active plan candidate:
  no_delta
else if multiple plan deliverables were touched in this turn:
  no_delta + planner checkpoint
else if no plan evidence bundle:
  no_delta
else if active plan deliverable not touched and blocker not cleared:
  no_delta
else:
  success
```

This keeps v2 convergence discipline while removing LLM field fragility and reducing false plan binding.

---

## 6. Ownership Model for Turn Files

The fixed files are runtime-owned, not LLM-authored:

- `cmd.txt`: runtime snapshot from last shell/tool execution
- `stdout.txt`: runtime-captured stdout
- `stderr.txt`: runtime-captured stderr
- `exit_code.txt`: runtime-captured exit code
- `result.json`: runtime-written final classification payload
- `artifacts/tool-events.jsonl`: runtime append-only tool event log
- `artifacts/changed_files.json`: runtime workspace write snapshot
- `artifacts/patch.diff`: runtime repository diff snapshot (when applicable)

Hard rule: fixed files are created every turn, even if no tool calls happened (empty content allowed).

LLM never writes these files directly.
LLM can only request tool actions; runtime records outcomes.

## 7. Concrete Example (How Auto-Generation Works)

Turn `0016` executes:

1. tool call: `bash` -> `pytest -q`
2. tool call: edit repo file `src/evaluator.ts`
3. tool call: write `runs/turn-0016/artifacts/test-summary.md`

Runtime auto-generates:

- `runs/turn-0016/cmd.txt`
- `runs/turn-0016/stdout.txt`
- `runs/turn-0016/stderr.txt`
- `runs/turn-0016/exit_code.txt`
- `runs/turn-0016/artifacts/tool-events.jsonl`
- `runs/turn-0016/artifacts/changed_files.json`
- `runs/turn-0016/artifacts/patch.diff`
- `runs/turn-0016/result.json`

Post-turn attribution then binds plan:

- `touched_files` matches `P3 deliverable: test-summary.md`
- current `ACTIVE` is `P2`, but attribution chooses `P3` because deliverable match is stronger
- `statusChange = P3 TODO -> ACTIVE` (or `P3 ACTIVE -> DONE` if cumulative done rules are met)

If both `P2` and `P3` deliverables are touched in one turn, runtime returns `no_delta` and triggers planner checkpoint for split execution.

---

## 8. Planner Checkpoint Boundary (Unchanged but Enforced in Runtime)

Non-checkpoint turns:

- allowed: evidence/facts/constraints/claims append
- forbidden: plan structure rewrite, drop/merge/reorder, done_definition edits

Checkpoint turns:

- allow controlled plan structure edits with validation

This prevents plan churn while preserving replan capability.

---

## 9. Migration Plan

1. Phase 1 (dual-write compatibility):
   keep reading legacy LLM control fields, but runtime-derived values take priority.
2. Phase 2 (runtime authoritative):
   stop requiring LLM control fields in schema validation.
3. Phase 3 (cleanup):
   delete prompt instructions and repair loops for LLM-authored control metadata.

No PROJECT.md or runs layout migration is required.

---

## 10. Acceptance Criteria

1. LLM output without `activePlanId/statusChange/delta/evidencePaths` still produces valid turn execution and `result.json`.
2. Missing business artifact write does not corrupt turn bookkeeping; fixed runtime files are still complete.
3. `success` cannot be produced unless runtime detects active plan deliverable touch or blocker clear.
4. Repeated compile/test-only turns without deliverable touch converge to `no_delta` and trigger planner checkpoint/blocker escalation.
5. When touched files match a non-ACTIVE plan deliverable, runtime binds to that plan (post-turn attribution).
6. When touched files match multiple plan deliverables in one turn, runtime returns `no_delta` and requests split.
7. Facts/claims without explicit evidence refs are auto-linked to turn evidence bundle or downgraded to `[HYP]`.
8. Existing path contract (`runs/turn-xxxx/...`) remains strictly enforced.

---

## 11. Tradeoffs

Pros:

- much smaller LLM schema surface
- fewer false `no_delta` from malformed output fields
- deterministic, testable control-plane behavior
- stronger evidence linkage without LLM path writing

Cons:

- runtime logic becomes the single complexity center
- requires reliable tool event and workspace diff capture

This tradeoff is intentional: move complexity from probabilistic output parsing to deterministic code.
