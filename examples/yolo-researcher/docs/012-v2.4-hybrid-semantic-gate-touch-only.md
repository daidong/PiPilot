# YOLO-Researcher v2.4: Hybrid Semantic Gate (Touch-Only) RFC-012

> Status: implemented (touch-only), rollout in progress.
> Scope: reduce false `NO_DELTA: missing_plan_deliverable_touch` while preserving deterministic safety boundaries.

---

## 0. Context

Current runtime already has strong deterministic gates in `examples/yolo-researcher/v2/session.ts`.

Observed recurring issue:

1. turns perform real work and produce artifacts
2. deterministic matching cannot prove deliverable touch
3. turn downgraded to `NO_DELTA: missing_plan_deliverable_touch`

This RFC introduces a constrained semantic gate to correct this specific false-negative path.

---

## 1. Decision Summary

### 1.1 What We Will Do

Use a hybrid gate:

1. deterministic gates stay canonical
2. LLM is used only for semantic attribution in one narrow case
3. semantic decision must be proof-carrying and mechanically verifiable

### 1.2 What We Will Not Do

1. no full LLM replacement of runtime gates
2. no direct `no_delta -> success` by semantic gate
3. no override of hard violations (path/schema/policy/coding-large-repo constraints)
4. no semantic downgrade path in this phase (LLM cannot turn success into no_delta)

---

## 2. Goals and Non-Goals

## 2.1 Goals

1. reduce false negatives for `missing_plan_deliverable_touch`
2. maintain reproducibility, replayability, and auditability
3. keep rollout low-risk via feature flags and shadow mode

## 2.2 Non-Goals

1. broad semantic judging of all statuses
2. replacing done-definition mechanics with free-form LLM judgement
3. changing existing hard gate semantics

---

## 3. Design Constraints (Hard)

1. Runtime deterministic checks execute first and remain source-of-truth.
2. Semantic gate can only participate when deterministic status would be `no_delta` with reason `missing_plan_deliverable_touch`.
3. Semantic output requires evidence references that pass mechanical validation.
4. Semantic gate must support abstention (`abstain`).
5. All semantic decisions must be replayable from persisted inputs and metadata.

---

## 4. Runtime Flow (Proposed)

Inside `runNativeTurn` in `examples/yolo-researcher/v2/session.ts`, after deterministic plan checks and before final status persistence:

1. compute deterministic outcome as today
2. if not eligible trigger condition, skip semantic gate
3. if eligible, call `semanticGateEvaluate()`
4. validate semantic output mechanically
5. if accepted, mark `deliverableTouched=true` for attribution purpose
6. re-run existing deterministic status pipeline (do not force success)

This keeps semantic gate as attribution correction, not status authority.

---

## 5. Eligibility Trigger

Semantic gate is invoked only when all are true:

1. deterministic `finalStatus === 'no_delta'`
2. `blockedReason === 'missing_plan_deliverable_touch'`
3. no hard violation flags are present
4. there is concrete work evidence in this turn, at least one:
   - `changed_files.json` non-empty
   - `patch.diff` exists
   - business artifact evidence exists
   - non-empty command output with successful execution

If any condition fails, semantic gate is skipped.

---

## 6. Semantic Gate Input Contract

Input must be normalized, bounded, and injection-resistant.

## 6.1 Input Object

```json
{
  "schema": "yolo.semantic_gate.input.v1",
  "turn": {"id": "turn-0021", "number": 21},
  "active_plan_id": "P3",
  "deterministic": {
    "status": "no_delta",
    "blocked_reason": "missing_plan_deliverable_touch"
  },
  "plan": {
    "done_definition": ["deliverable: artifacts/benchmark_results.json", "evidence_min: 1"],
    "deliverables": ["artifacts/benchmark_results.json"]
  },
  "evidence_summary": {
    "explicit_evidence_paths": ["runs/turn-0021/..."],
    "business_artifacts": ["runs/turn-0021/artifacts/..."],
    "workspace_write_touches": ["runs/turn-0021/artifacts/..."],
    "changed_files_count": 3,
    "has_patch": true,
    "cmd_exit_code": 0
  },
  "repo_constraints": {
    "hard_violations": [],
    "coding_large_repo_required": false
  }
}
```

## 6.2 Input Safety Rules

1. do not pass raw long stdout/patch bodies
2. pass structured summaries only (counts, paths, bounded snippets)
3. canonical sort all arrays before hashing
4. fixed truncation strategy and max payload size

---

## 7. Semantic Gate Output Contract

```json
{
  "schema": "yolo.semantic_gate.output.v1",
  "verdict": "touched|not_touched|abstain",
  "confidence": 0.0,
  "touched_deliverables": [
    {
      "id": "artifacts/benchmark_results.json",
      "evidence_refs": ["runs/turn-0021/artifacts/benchmark_results.json"],
      "reason_codes": ["file_created", "deliverable_semantic_match"]
    }
  ],
  "notes": "optional short note"
}
```

Validation requirements:

1. `verdict` must be one of `touched|not_touched|abstain`
2. `confidence` in `[0,1]`
3. if `verdict=touched`, at least one touched item and non-empty `evidence_refs`
4. each evidence ref must exist and be a canonical `runs/turn-xxxx/...` path

---

## 8. Arbiter Rules (Touch-Only)

## 8.1 Acceptance Conditions

Semantic correction is accepted only if all are true:

1. mode allows enforcement
2. verdict is `touched`
3. confidence >= threshold (default `0.85`)
4. evidence ref validation passes
5. touched deliverable intersects active plan deliverable set

## 8.2 Effect of Acceptance

1. mark plan deliverable as touched for this turn attribution
2. re-evaluate existing deterministic status logic
3. do not directly assign success

## 8.3 Explicitly Forbidden

1. semantic gate cannot override hard violations
2. semantic gate cannot perform downgrade in this phase

---

## 9. Feature Flags and Modes

Add semantic gate config:

```ts
semanticGate?: {
  mode?: 'off' | 'shadow' | 'enforce_touch_only' | 'enforce_success'
  confidenceThreshold?: number
  model?: string
  maxInputChars?: number
}
```

Mode behavior:

1. `off`: no call
2. `shadow`: call + log, no status impact
3. `enforce_touch_only`: allow only touch attribution correction
4. `enforce_success`: reserved for future, disabled by default

---

## 10. Replayability and Audit Fields

Persist to `runs/turn-xxxx/result.json` under `semantic_gate`:

1. `enabled`, `mode`
2. `invoked`, `eligible`
3. `model_id`, `prompt_version`, `temperature` (must be `0`)
4. `input_hash`
5. `output` (structured)
6. `accepted` and `reject_reason` if rejected

This allows deterministic post-mortem and offline replay.

## 12. Pseudocode

```ts
if (finalStatus === 'no_delta' && blockedReason === 'missing_plan_deliverable_touch' && eligibleForSemanticGate()) {
  const sg = await semanticGateEvaluate(composeSemanticInput())
  const accepted = shouldAcceptSemanticTouch(sg, activePlanDeliverables)

  recordSemanticGateTelemetry(sg, accepted)

  if (mode === 'enforce_touch_only' && accepted) {
    doneDefinitionCheck = {
      ...doneDefinitionCheck,
      deliverableTouched: true
    }

    // Re-run existing deterministic status reconciliation.
    recomputeFinalStatusFromExistingRules()
  }
}
```

## 13. Test Strategy

Required tests:

1. shadow mode logs but does not affect status
2. enforce_touch_only corrects eligible false no_delta
3. hard violation present -> semantic gate cannot change outcome
4. invalid evidence refs -> semantic output rejected
5. abstain -> no change
6. low confidence -> no change
7. touched verdict without deliverable intersection -> no change

---

## 14. Rollout Plan

### Phase A: Shadow

1. enable `shadow`
2. run >= 100 turns
3. collect confusion matrix against manual sample labels

### Phase B: Enforce Touch-Only

1. enable `enforce_touch_only` for limited runs
2. watch false-positive rate and downstream DONE integrity

### Phase C: Decide on Expansion

Only if metrics are stable, discuss broader semantic roles.

---

## 15. Success Metrics

1. reduction in `missing_plan_deliverable_touch` no_delta rate
2. no increase in hard-violation bypass incidents (must remain zero)
3. stable or improved turns-to-DONE
4. verification-backed success ratio not degraded
