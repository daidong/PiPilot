# YOLO-Researcher v2.5: Deterministic Artifact Anchor (RFC-013)

> Goal: eliminate path-anchor drift (for example `openevolve_repo/runs/...`) while preserving repo-local execution.
> This revision is aligned to the post-refactor code split (`session.ts` + `session-native-turn.ts`).

---

## 0. Post-Refactor Baseline (Current Code)

As of the current refactor, the runtime already has strong path discipline primitives:

1. Canonical turn root is created by session runtime:
   - `runs/turn-xxxx/`
   - `runs/turn-xxxx/artifacts/`
2. Evidence pointer normalization + validation exists (`runs/turn-xxxx/...` contract).
3. Runtime-derived control plane is active:
   - `result.json` includes runtime-derived plan attribution/status/delta fields.
4. Code-touch governance exists:
   - repo code edits without `coding-large-repo` workflow can be downgraded.

Current module ownership after refactor:

- `examples/yolo-researcher/v2/session.ts`
  - orchestration, path normalization, evidence repair, workspace diff artifacts
- `examples/yolo-researcher/v2/session-native-turn.ts`
  - native-turn mutations, status guards, result payload builder
- `examples/yolo-researcher/v2/llm-agent.ts`
  - turn prompt + `yolo.turnArtifactsDir` session state injection
- `examples/yolo-researcher/v2/tool-wrappers.ts`
  - wrapper output-dir coercion into current turn artifacts

---

## 1. Remaining Failure Mode

Even with the baseline above, drift can still happen when:

1. execution runs in nested repo `cwd` (expected), and
2. tool/subagent uses relative output `runs/...` resolved from that nested `cwd` (unexpected for evidence contract).

Result:

- artifacts may land in nested `*/runs/turn-*`
- canonical deliverable touch is missed
- turns degrade to `no_delta` despite real work

---

## 2. Decision (v2.5 Target)

1. Keep execution `cwd` behavior unchanged (repo-local execution is required).
2. Make artifact anchor runtime-owned and deterministic per turn.
3. Introduce explicit repo targeting for code-touch operations (`repo_id` or explicit repo `cwd`), no implicit default repo.
4. Prefer logical artifact URI (`artifact://...`) over hand-written `runs/...` in prompts/subagents.
5. Add turn-end path-anchor audit for nested/non-canonical `runs/turn-*` writes.
6. Support policy modes:
   - `recover` for interactive sessions
   - `fail` for CI/strict sessions
7. Expose path-anchor telemetry in `result.json` and EvidenceView.

---

## 3. Non-Goals

1. Multi-root workspace whitelisting.
2. Replacing all existing skill scripts in one pass.
3. Breaking repo-local command execution (`git`, `pytest`, `codex -C ...`).

This RFC assumes single workspace root with potentially many nested repos/versions.

---

## 4. Target Model

### 4.1 Two Coordinates (Must Stay Separate)

1. `exec_cwd`: where command/tool executes.
2. `artifact_anchor`: where turn evidence is stored.

Canonical artifact anchor for turn `N`:

- `TURN_ID = turn-${String(N).padStart(4, '0')}`
- `TURN_ARTIFACTS_REL = runs/${TURN_ID}/artifacts`
- `TURN_ARTIFACTS_ABS = <workspace>/${TURN_ARTIFACTS_REL}`

### 4.2 Repository Targeting (Single-Root Multi-Repo)

For code-touch actions:

1. runtime resolves target from `repo_id` (preferred) or explicit repo `cwd`
2. missing/ambiguous target is policy violation
3. resolved target is written to turn telemetry

### 4.3 Artifact URI

Use logical URI in prompts/subagents:

- `artifact://<relative_path>`

Runtime resolves to `TURN_ARTIFACTS_ABS/<relative_path>`.

The following are not fully implemented yet and are the v2.5 delta:

1. `skill-script-run` env injection for anchor metadata:
   - `AF_WORKSPACE_ROOT`
   - `AF_TURN_ID`
   - `AF_TURN_ARTIFACTS_REL`
   - `AF_TURN_ARTIFACTS_ABS`
2. `artifact://` resolver in wrappers/prompt conventions.
3. explicit `repo_id` registry + enforcement for code-touch turns.
4. turn-end nested-runs audit + rewrite/fail policy.
5. path-anchor telemetry fields and UI rendering.

Related but already present (do not duplicate work):

1. evidence-path normalization and repair utilities in `session.ts`
2. runtime result payload assembly in `session-native-turn.ts`
3. wrapper-level canonical outputDir coercion in `tool-wrappers.ts`

## 7. Result Payload Additions (Target)

Add fields in `result.json`:

1. `resolved_repo: { repo_id, repo_path }` for code-touch turns
2. `path_anchor_violation: { detected, count, samples[] }`
3. `path_rewrite_events: [{ from, to, reason }]`
4. `path_anchor_metrics: { scanned_paths, nested_runs_count, rewritten_count, mode }`

These are additive to existing runtime-derived fields.

---

## 8. Policy Modes

1. `recover`
- copy misplaced artifacts into canonical turn evidence path
- record rewrite events
- continue turn classification

2. `fail`
- classify turn as failure/blocked with explicit anchor violation
- no silent continuation

Recommended defaults:

- interactive/dev: `recover`
- CI/strict governance: `fail`

Feature flags:

- `YOLO_PATH_ANCHOR_AUDIT=1`
- `YOLO_PATH_ANCHOR_MODE=recover|fail`
- `YOLO_REQUIRE_REPO_TARGET=1`
- `YOLO_ARTIFACT_URI_PREFERRED=1`

## 9. Test Plan (Aligned to Current Test Layout)

Primary files:

- `tests/tools/skill-script-run.test.ts`
- `tests/yolo-researcher-v2/runtime-contract.test.ts`

Required cases:

1. nested repo `cwd` + relative output request still lands evidence in canonical workspace turn artifacts.
2. nested `*/runs/turn-*` is detected as anchor violation.
3. `recover` mode rewrites misplaced artifact and logs rewrite record.
4. `fail` mode blocks turn with explicit violation reason.
5. code-touch turn without `repo_id`/explicit repo `cwd` is rejected deterministically.
6. code-touch turn with `repo_id` resolves correctly and is persisted in `result.json`.
7. `artifact://` outputs resolve correctly across nested repo execution.

---

## 10. Rollout

1. Phase 0 (current baseline): keep behavior, document ownership after refactor.
2. Phase 1: add `AF_*` env injection + audit telemetry (no enforcement).
3. Phase 2: enable `artifact://` in wrappers/prompts + compatibility fallback.
4. Phase 3: enable `recover` by default in interactive sessions.
5. Phase 4: enable `fail` by default in CI/strict sessions + mandatory repo target policy.

---

## 11. Acceptance Criteria

RFC is complete when:

1. repo-local execution remains functional with explicit `cwd`.
2. artifact paths are deterministic under workspace-root turn artifacts.
3. nested non-canonical turn outputs are never silent.
4. code-touch actions are deterministically bound to explicit repo target.
5. EvidenceView exposes path-anchor governance state.
6. regression tests cover the original drift pattern and new enforcement.

---

## 12. Open Questions

1. Should rewrite preserve source tree (`recovered/<repo>/...`) or flatten names?
2. Should `repo_id` be mandatory only when multiple repos are discovered?
3. Should `artifact://` become hard-required in strict mode after migration window?
