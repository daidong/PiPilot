# YOLO-Researcher v2.6: Complexity Reduction Roadmap (RFC-014)

Status: Proposed  
Owner: yolo-researcher/v2  
Scope: `examples/yolo-researcher/v2` + shared runtime core  
Type: Refactor (no behavior change by default)

## 1. 背景与问题

当前系统可用，但复杂度已经开始拖慢迭代速度，主要表现为：

1. 超长方法和混合职责导致改动风险高（一个修改容易触发连锁影响）。
2. 规则逻辑、I/O 落盘、状态机、恢复策略在同一层混写，测试虽可覆盖但认知负担过大。
3. 部分新拆分模块类型边界仍弱（`any` 过多），后续维护易出现“能跑但不可证明正确”。

这不是功能缺失，而是工程复杂度债务。

## 2. 目标

RFC-014 的目标是“降复杂度，不改语义”：

1. 降低单文件/单函数复杂度，提升可读性与可验证性。
2. 强化类型边界，减少 `any` 与隐式契约。
3. 保持当前行为一致（以现有测试与回归日志为准）。
4. 为后续功能（semantic gate、executing-plans、runtime gate）提供稳定承载结构。

## 3. 非目标

1. 不在本 RFC 中引入新产品能力。
2. 不在本 RFC 中修改 gate 判定策略语义。
3. 不在本 RFC 中扩大工具权限或变更安全策略。

## 4. 优先重构对象（按收益/风险排序）

### P0（最高优先）

1. `src/agent/agent-loop.ts` 的 `run()` 大方法拆分。
2. `examples/yolo-researcher/v2/session-native-turn.ts` 的类型强化和门禁拆段。

### P1

1. `examples/yolo-researcher/v2/llm-agent.ts` 的 `runTurn()` 拆分。
2. `examples/yolo-researcher/v2/project-store.ts` 的 `applyUpdate/applyTurnPlanDelta` 规则层抽离。

### P2

1. `examples/yolo-researcher/v2/tool-wrappers.ts` 的 `createYoloToolWrapperPack()` 分域拆分。

## 5. 具体实施计划

## Phase A: Agent Loop Pipeline 化（P0）

目标文件：`src/agent/agent-loop.ts`

拆分策略：

1. 提取 `prepareRound()`：轮次工具集合、roundHint、消息装配。
2. 提取 `executeRoundLLM()`：一次模型调用、usage 回调、输出标准化。
3. 提取 `executeToolBatch()`：工具调用、错误分类、retry/strike 累积。
4. 提取 `finalizeRun()`：budget summary、return payload 收敛。

验收标准：

1. `run()` 主体长度显著下降（目标 < 300 行）。
2. 原有测试全部通过。
3. trace 事件字段不变（名称和关键字段兼容）。

---

## Phase B: Native Turn Guard Typed 化（P0）

目标文件：`examples/yolo-researcher/v2/session-native-turn.ts`

拆分策略：

1. 将 `applyNativeTurnStatusGuards` 拆为：
   - `applyHardGuards`
   - `applySemanticTouchArbitration`
   - `applyRedundancyBlockGuards`
2. 给导出方法补齐强类型输入/输出（去掉 `any`）。
3. 将“状态变更”与“summary 文案拼接”分离。

验收标准：

1. 对外 API 只有明确 typed contract。
2. semantic gate 相关字段在 result.json 里保持不变。
3. `tests/yolo-researcher-v2/runtime-contract.test.ts` 全绿。

---

## Phase C: LLM Turn Orchestration 拆分（P1）

目标文件：`examples/yolo-researcher/v2/llm-agent.ts`

拆分策略：

1. 提取 `runSingleAttempt()`：单轮 agent.run + parse + outcome normalize。
2. 提取 `handleAttemptFailures()`：runtime failure / policy-block / bootstrap failure 的统一分支。
3. 提取 `buildNextPromptAfterFailure()`：repair prompt 选择逻辑集中。

验收标准：

1. `runTurn()` 仅保留流程骨架。
2. ask_user 升级条件保持一致。
3. 现有行为测试不回归。

---

## Phase D: Project Store 规则层抽离（P1）

目标文件：`examples/yolo-researcher/v2/project-store.ts`

拆分策略：

1. 引入纯函数规则模块（建议 `project-store-rules.ts`）：
   - `computeNextPlanBoard()`
   - `validateTurnPlanDelta()`
   - `mergePanelUpdate()`
2. `ProjectStore` 保留 IO（load/save）和 orchestrate。

验收标准：

1. 规则函数可单测（无文件系统依赖）。
2. 计划状态迁移语义一致。

---

## Phase E: Tool Wrapper 分域（P2）

目标文件：`examples/yolo-researcher/v2/tool-wrappers.ts`

拆分策略：

1. 以职责拆成：
   - `tool-wrappers-fs.ts`
   - `tool-wrappers-git.ts`
   - `tool-wrappers-skill.ts`
   - `tool-wrappers-literature.ts`
2. 保持统一入口 `createYoloToolWrapperPack()`，内部组合子模块。

验收标准：

1. 对外导出签名不变。
2. 现有工具行为与事件记录兼容。

## 6. 质量门禁（每个 Phase 必须满足）

1. TypeScript build 通过：`npm run build`
2. YOLO v2 runtime contract 通过：
   - `npm run test:run -- tests/yolo-researcher-v2/runtime-contract.test.ts`
3. 若触及 Agent Loop，再补：
   - `npm run test:run -- tests/core/errors.test.ts tests/core/retry.test.ts tests/core/feedback.test.ts`
4. 每个 Phase 单独提交（可回滚）。

## 7. 风险与缓解

风险：

1. 重构引入隐性行为漂移（尤其是状态机和 summary 文案）。
2. 过度抽象导致调用链过深，排查成本上升。

缓解：

1. 先“提取函数”，后“迁文件”；每步都跑回归。
2. 对关键行为保留 golden 样例（NO_DELTA/ask_user/blocked 路径）。
3. 采用小步提交策略（每 phase 一到两个 commit）。

## 8. 里程碑与退出条件

里程碑：

1. M1: Phase A+B 完成并稳定。
2. M2: Phase C+D 完成。
3. M3: Phase E 完成，文档更新。

退出条件：

1. 上述 Phase 全部完成且测试全绿。
2. 关键主函数复杂度显著下降（见附录阈值）。

## 9. 附录：复杂度目标阈值

建议阈值（非硬性，但应趋近）：

1. 单方法 > 200 行：必须拆分。
2. 单文件 > 1500 行：应考虑模块化拆分。
3. 关键 orchestrator 函数（run/turn）控制在 200-350 行骨架。
4. 新增 `any` 零容忍（除跨边界临时适配层）。
