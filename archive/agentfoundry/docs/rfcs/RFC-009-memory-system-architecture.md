# RFC-009: Memory System Architecture Redesign (Project Cards + WorkingSet)

**Status**: Draft
**Author**: AI Agent Design Team
**Created**: 2026-02-04
**Target**: AgentFoundry Framework + Research Pilot + Personal Assistant

---

## Abstract

This RFC proposes a framework-level redesign of memory to remove overloaded pin semantics and make long-running projects stable without blowing context budgets. The design introduces a single persistent concept — **Project Cards** — and a single runtime concept — **WorkingSet Cards** — plus session history. Project Cards capture evolving core decisions and constraints. WorkingSet Cards are assembled per request from explicit selection, working-set continuity, and retrieval. Context budgets are enforced via shape degradation (full → excerpt → card → index-line). Session memory is integrated into StateSummary instead of bypassing budgets.

---

## Problem Summary

Current memory behavior has multiple issues:

1. **Overloaded pinning**: “Pinned” is used for UI, long-term memory, and always-in-context behavior.
2. **Long-term drift**: Important project decisions evolve, but pinned items are treated as static anchors.
3. **Budget bypass**: Session memory is injected as a message prefix, ignoring context budgets.
4. **Shape sprawl**: Summary, excerpt, and full forms are generated by separate mechanisms without a single degradation ladder.
5. **Selection ambiguity**: `selectedForAI` persists per entity even though selection is a per-request runtime concern.

---

## Design Principles

1. **Canonical Authority**: Disk entities are the single source of truth.
2. **Project Cards = Long-Term Alignment**: Persistent cards represent core decisions and constraints, and can be updated over time.
3. **WorkingSet = Runtime Only**: WorkingSet Cards are computed each request; they are not persisted in entities.
4. **Budget First**: Context inclusion is always budgeted and shape-degraded, never bypassed.
5. **Explainable Inclusion**: Every included item has an explicit reason.

export interface MemoryEntity {

- **Remove** `PinPolicy` and `selectedForAI`.
- **Add** `projectCard` (single persistent boolean).
- **WorkingSet** becomes runtime-only and derived per request.

## Phase 3: Project Cards Semantics

### 3.1 Definition

- A **Project Card** is any entity with `projectCard=true`.
- Project Cards represent **core decisions, constraints, and alignment**.
- They are **not limited in count**. Budget pressure is handled by shape degradation, not by count limits.

### 3.2 Default Behavior

| Tool | projectCard default | Notes |
|------|---------------------|-------|
| `save-note` | false | user can set `projectCard=true` explicitly |
| `save-paper` | false | promotes only when explicitly flagged |
| `save-data` | false | promotes only when explicitly flagged |

### 3.3 Promotion API (No New Tool Required)

- Use existing `save-*` and `update-*` to set `projectCard=true`.
- UI may expose a “Mark as Project Card” toggle using `update-*`.

### 3.4 Guidance on Count & Staleness (Soft Policy)

- **Typical active count**: 5–15 Project Cards per project. This is guidance only, not an enforced limit.
- **Why guidance**: Large counts are supported via shape degradation, but many `index-line` cards still consume budget.
- **Review cues**: Apps should surface a “Review Project Cards” prompt when:
  - `projectCard` count exceeds a configurable threshold (default 20), or
  - a card has not been referenced or updated in a long period (default 30–90 days, app-configurable).
- **No auto-demotion by default**: Demotion should be user-confirmed to avoid hidden behavioral changes.

### 3.5 Project Card Evolution

- Project Cards can be updated via `update-*` with normal revision increments.
- Agents may **propose** edits, but apps should require user confirmation for changes tagged as “core decision/constraint.”
- Recommended metadata:
  - `updatedAt` already exists for recency.
  - Optional app-level history log (e.g., `project-card-history.jsonl`) for auditability.

## Phase 4: WorkingSet Builder (Runtime)

WorkingSet Cards are assembled per request from multiple signals. They are never persisted.

### 4.1 Sources (in priority order)

1. **Explicit**: `@mention` or UI “Add to Working Set” in the current session.
2. **Continuity**: recently used entities in this session.
3. **Retrieval**: relevance search over titles, tags, and summary cards.
4. **Index Hints**: low-cost index lines for “potentially useful” items.

### 4.2 Continuity Rules (Precise)

- **Definition of “use”**:
  - Explicit mention (`@mention` or UI add)
  - Tool access (`ctx-expand` or explicit read of the entity)
  - Entity update (`update-*`)
- **Window**: last N turns (default 6) or last T minutes (default 30). App-configurable.
- **Decay**: score decays by turn distance; continuity is a boost, not a guarantee.

### 4.3 Retrieval Scope

- **Default**: search over `title + tags + summaryCard`.
- **Optional deep retrieval**: apps may enable canonical content search when summaries are insufficient.
- **Budget-safe fallback**: when a candidate is found only via deep retrieval, include it as `index-line` and rely on `ctx-expand` for full content.

### 4.4 WorkingSet Plan

```ts
export interface WorkingSetItem {
  entityId: string
  source: 'explicit' | 'continuity' | 'retrieval' | 'index'
  requestedShape: 'full' | 'excerpt' | 'card' | 'index-line'
  relevanceScore: number
  reason: string
}
```

### 4.5 Budget Fitting

- Use a single degradation ladder for **both** Project Cards and WorkingSet:
  `full → excerpt → card → index-line → drop`.
- This guarantees stable budgets without hard count limits.

## Phase 5: StateSummary (Session Memory Integration)

Session memory must be budgeted and assembled in the `state-summary` phase.

```ts
const SESSION_MEMORY_CONFIG = {
  ttl: 2 * 60 * 60 * 1000,  // 2 hours sliding
  maxItems: 8,
  maxTokens: 600,
  refreshOnRead: true,
}
```

Remove any message-prefix injection of session memory.

**Config note**: `ttl` and `maxTokens` must be app-configurable. For long research workflows, apps may choose longer TTLs or explicit session-boundaries rather than time-based expiry.

## Phase 6: Project Brief Card (Optional)

A “Project Brief” is an optional Project Card auto-generated or user-authored. It replaces the special “always pinned dashboard” concept and is treated the same as other Project Cards (subject to shape degradation).

## Phase 7: Dedup Mechanism (Minimal)

- **Exact hash** only (no similarity heuristics in core path).
- Warn but allow creation.

## Phase 8: Migration

### 8.1 Mapping Rules

| Old Field | New Field | Notes |
|----------|-----------|-------|
| `pinned=true` | `projectCard=true` | preserve long-term intent |
| `selectedForAI` | dropped | selection is runtime-only |

### 8.2 Migration Behavior

- On first load, entities are migrated in place.
- Deterministic `summaryCard` is generated if missing.

### 8.3 Migration Review UX (Recommended)

- Legacy `pinned=true` does not necessarily imply “core decision.” Apps should offer a review step.
- Suggested behavior:
  - Migrate `pinned → projectCard`.
  - Mark legacy-derived cards with a review flag (app-level metadata).
  - UI prompts user to confirm or demote to normal entities.

## Phase 9: Degradation Ladder + Budgets

### 9.1 Shape Ladder

| Level | Trigger | Shape Max | Actions |
|-------|---------|-----------|---------|
| L0 | usage < 70% | `full` | Normal |
| L1 | ≥ 80% | `excerpt` | Downgrade shapes |
| L2 | ≥ 95% | `card` | Drop low-priority WorkingSet |
| L3 | overflow | `index-line` | Only Project Cards + last message + tools |

### 9.2 Budget Profile (Example)

```ts
research: {
  project:  { min: 600, max: 4000, weight: 3 },  // Project Cards
  working:  { min: 0, maxPct: 0.30, weight: 4 }, // WorkingSet
  session:  { min: 800, max: 2000, weight: 2 },  // StateSummary
  historyIndex: { min: 500, max: 2000, weight: 1 }
}
```

## System Prompt Impact (Required)

### Summary

**Impact: Medium**. The core model can function without prompt changes, but existing prompts explicitly instruct auto-pinning and “pinned = always in context,” which conflicts with the new Project Cards + WorkingSet semantics. If not updated, the agent will over-create Project Cards and incorrectly assume persistence or inclusion rules.

### Required Prompt Updates (App-Level)

**Research Pilot**
- Remove claims like “Every note you create is automatically pinned and visible in every turn.”
- Replace with: “Only Project Cards are long-term. Use them for stable decisions/constraints; do not use for transient facts.”
- Add: “WorkingSet is assembled per request; use `@mention` or explicit selection to force inclusion.”
- Clarify session memory: “Session memory is short-term and budgeted in StateSummary; do not rely on prefix injection.”

**Personal Assistant**
- Remove/replace references to “pinned notes are always included.”
- Update “select” instructions to “WorkingSet,” clarifying that it is session-scoped and not persisted.
- Ensure tools/instructions match new commands (`/project`, `/workingset` or equivalent UI toggles).

### Optional Prompt Enhancements

- Add a short “Memory Rules” block:
  - Project Cards = long-term alignment.
  - WorkingSet = current decision context.
  - Session History = continuity only.

### Risk If Unchanged

- Over-promotion into Project Cards, causing long-term noise.
- False assumption that all Project Cards are always injected, leading to missing context under budget pressure.
- Confusion between runtime selection and persistent memory.

## Decision Rationale (Addressing External Review)

This section records rationale and responses to external feedback.

### 1. Project Card Count Pressure

**Concern**: Unlimited Project Cards could exhaust budget even at `index-line` level.  
**Decision**: Keep no hard limit (to support long projects), but add **soft guidance + review prompts**.  
**Rationale**: Hard limits force destructive behavior. Soft guidance keeps flexibility while protecting budget.\n

### 2. WorkingSet Continuity Definition

**Concern**: “Recently used” is vague; unclear triggers and decay.  
**Decision**: Define “use” explicitly (mention, tool access, update), add a configurable window and decay.  
**Rationale**: Continuity must be deterministic to avoid non-reproducible context.\n

### 3. Retrieval Quality vs Summary-Only Search

**Concern**: Summaries may omit critical details.  
**Decision**: Default to summary-based retrieval but allow optional deep retrieval over canonical content, with `index-line` fallback.  
**Rationale**: Balances recall quality against compute and budget safety; deep retrieval is app-configurable.\n

### 4. Migration Risk (Pinned ≠ Core)

**Concern**: `pinned → projectCard` may over-promote.  
**Decision**: Preserve mapping for compatibility but add a **review UX** flag for legacy-derived cards.  
**Rationale**: Avoids breaking behavior while offering a safe correction path.\n

### 5. Session Memory TTL

**Concern**: 2 hours is arbitrary and may be wrong for research workflows.  
**Decision**: Make TTL configurable per app; allow explicit session boundaries as alternative.  
**Rationale**: Session scope differs by product; the framework should not enforce a single policy.\n

### 6. Project Card Evolution

**Concern**: No update semantics for evolving core decisions.  
**Decision**: Allow updates via `update-*`; require user confirmation for “core decision/constraint” edits; optional history log.  
**Rationale**: Prevents silent drift while keeping updates lightweight and auditable.\n

## Decision Summary

| Dimension | Decision |
|-----------|----------|
| Long-term memory | Project Cards (`projectCard=true`) |
| Runtime memory | WorkingSet Cards (derived, not persisted) |
| Pin semantics | Removed (replaced by Project Cards) |
| Summary cards | Deterministic first, LLM fallback |
| Session memory | StateSummary phase only |
| Shape control | Single degradation ladder |
| Dedup | Exact hash only |
